# `siso_maxlogmap` Internal Control Functional Guide

This note decomposes the **updated** control logic inside [`rtl/siso_maxlogmap.vhd`](../../../rtl/siso_maxlogmap.vhd).

This is the rolling-window control version, not the older alpha-seed-prepass controller.

## 1. What Counts As The Control Unit Now

The control unit is the combination of:
- state register `st`
- segment registers:
  - `pair_cnt`
  - `seg_bits_i`
  - `win_cnt`
- traversal registers:
  - `load_idx`
  - `work_win_idx`
  - `work_local_idx`
- rolling state registers:
  - `alpha_work`
  - `beta_work`
  - `next_alpha_seed_reg`
- fetch handshake registers:
  - `fetch_req_valid_q`
  - `fetch_req_pair_idx_q`
- output pulse registers:
  - `out_valid_q`
  - `done_q`

## 2. Main Functional Control Blocks

### 2.1 Segment Setup Unit

In `IDLE`, this block:
- clamps `seg_len`
- computes `pair_cnt`
- computes `win_cnt`
- selects the initial forward seed from `seg_first`

### 2.2 Optional Load Controller

In `LOAD`, this block:
- accepts `in_valid`
- writes the legacy segment memories
- tracks `load_idx`

This block is inactive in external-fetch mode.

### 2.3 Pair Fetch Controller

This is the major new control block.

It controls:
- `FWD_REQ` / `FWD_WAIT`
- `DUMMY_REQ` / `DUMMY_WAIT`

Functions:
- issue the required pair index
- wait for the response
- load the response into `fetched_*_q`

### 2.4 Rolling Window Traversal Controller

This block manages:
- current window index `work_win_idx`
- local pair index `work_local_idx`

Traversal policy:
- current window forward traversal: ascending
- next window dummy backward traversal: descending
- current window local backward emission: descending

### 2.5 Rolling Boundary Controller

This block replaces the old alpha-seed BRAM controller.

It decides:
- `alpha_work` start seed for the current window
- `beta_work` seed for the current window
- `next_alpha_seed_reg` carry to the next window

Architectural meaning:
- only one inter-window alpha boundary is kept
- beta boundaries are regenerated by dummy backward over the next window

### 2.6 Local Cache Write Controller

In `FWD_STEP`, this block writes:
- `alpha_local_mem`
- `gamma_local_mem`
- `g0_local_mem`
- `g1_local_mem`
- local `sys/apri` memories

### 2.7 Output Emit Controller

In `LOCAL_BWD`, this block:
- asserts `out_valid_q`
- drives `out_pair_idx_q`
- writes the outgoing `ext/post` pair
- decides whether to move to the next window or finish

## 3. Updated State Set

Current states:
- `IDLE`
- `LOAD`
- `FWD_REQ`
- `FWD_WAIT`
- `FWD_STEP`
- `DUMMY_REQ`
- `DUMMY_WAIT`
- `DUMMY_STEP`
- `LOCAL_BWD`
- `FINISH`

Removed states from the old architecture:
- `DUMMY_FWD`
- `FWD_SEEDS`
- `PREP_WINDOW`
- `LOAD_ALPHA_WAIT`
- `LOAD_ALPHA`
- `LOCAL_FWD`

## 4. Active State Sequence

### External-Fetch Mode

```text
IDLE
  -> FWD_REQ
  -> FWD_WAIT
  -> FWD_STEP
  -> DUMMY_REQ / DUMMY_WAIT / DUMMY_STEP
  -> LOCAL_BWD
  -> FWD_REQ for next window
  -> ...
  -> FINISH
```

### Legacy Load Mode

```text
IDLE
  -> LOAD
  -> FWD_REQ
  -> FWD_STEP
  -> DUMMY_REQ / DUMMY_STEP
  -> LOCAL_BWD
  -> ...
  -> FINISH
```

## 5. State Meaning

### `IDLE`
- admit a new segment
- choose legacy-load vs external-fetch entry

### `LOAD`
- fill the compatibility segment RAMs

### `FWD_REQ`
- request or read the next pair of the current window

### `FWD_WAIT`
- wait for fetch response in external mode

### `FWD_STEP`
- update local caches
- advance forward alpha
- either continue within the window or start beta-seed derivation

### `DUMMY_REQ`
- request or read the next pair of the following window

### `DUMMY_WAIT`
- wait for the next-window fetch response in external mode

### `DUMMY_STEP`
- run backward ACS on the following window
- when done, `beta_work` becomes the current-window beta seed

### `LOCAL_BWD`
- emit one output pair from the current window
- update backward metrics
- either continue in the current window or advance to the next window

### `FINISH`
- pulse `done`

## 6. Control-Policy Summary

The new control policy is:
- optional preload
- fetch one current-window pair
- compute/store one current-window forward step
- fetch one next-window pair
- compute one dummy backward step
- emit current-window outputs backward
- roll forward to the next window

So the current controller is:
- fetch-driven
- rolling-window
- forward-ordered across windows

not:
- segment-prepass
- seed-BRAM driven
- reverse-window ordered
